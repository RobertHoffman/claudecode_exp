量化策略设计文档在"语义层 + 流程层"完成后，必须**额外自查 4 类执行层冲突**——用户在 2026-06-25 review PR-N Dual-Track v1.7.4 时一次性识别出这 4 类问题。

**典型陷阱**（2026-06-25 用户专业级 review v1.7.4 一次性找出 4 处冲突）：

| # | 冲突类型 | v1.7.4 案例 | 修复 |
|---|---------|-----------|------|
| 1 | **日内时序左右互搏** | "1 分钟抢筹" vs "3-5 分钟观察期内禁止交易"——全自动脚本逻辑死锁 | **分时动量触发条件单**：09:31 突破立即市价单 / 横盘挂单 / 09:35 复核 |
| 2 | **技术指标语义错配** | "量新高"被写成 `vol >= MA(vol, 30d) * 1.5`（量能激增），不是 `vol == MAX(vol, 30d)`（绝对天量）| 文档与代码必须**字面对齐**：要么改代码用 MAX，要么改文档用"量能激增" |
| 3 | **数据总线时钟残留** | D 日 / T 日命名是 v1.3 时代，但 v1.7.4 引入顶部确立模型后时钟已位移 1 天 | **统一命名**：`index_top_date` / `topping_window` / `entry_date` / `holding_window` |
| 4 | **模糊常量缺数学判别式** | "大多数高开" 是人类自然语言，Agent 写代码会拍脑袋写 >50% | **强制量化**：`STOCK_TOPPING_HIGH_OPEN_RATIO = 0.60` + `MIN_DAYS = 3` 双条件兜底 |

**Why**：用户 review 时已经从"读者视角"转向"**自动化交易员视角**"——他不只看文档是否合理，更看**脚本能不能无歧义执行**。4 类冲突的共同特征是：**语义层都说得通，但工程层会逻辑死锁 / 语义错位 / 命名误导 / 编译失败**。

**How to apply**（设计文档 v1.7.4 → 实施前必须走 4 类执行层 checklist）：

1. **日内时序左右互搏自检**：
   - 列出所有时间点（09:30 / 09:31 / 09:35 / 09:30-09:35 / 09:35-10:00 等）的**触发动作**
   - 每个时间点必须**只能扮演一个角色**：硬门槛 / 软门 / 触发条件 / 复核点
   - 如果两条规则在同一时间窗口**互斥**（如"前 3 分钟禁止交易" + "1 分钟内抢筹"）→ **逻辑死锁**，必须拆为**条件单逻辑**：
     ```
     09:31:00 检查项：分时线是否突破 X%
       是 → 立即市价单（达成"1 分钟抢筹"）
       否 → 挂单等待（不强制"禁止交易"）
     09:35:00 复核项：5 分钟涨幅是否 ≥ X%
       是 → 吃尾单入场
       否 → 硬剔除
     ```
   - **核心原则**：用**条件触发**替代**主观观察**——不要写"前 X 分钟禁止交易"，要写"X 分钟后若条件 Y 满足则触发 Z"

2. **技术指标语义错配自检**（每条"语义名称" vs 对应公式）：
   - 列出所有用人类语言命名的指标（"量新高" / "价新高" / "趋势向上" / "强势" 等）
   - 每个名称必须**字面对应**一个**唯一**的工程公式
   - **常见错配**：
     - "量新高" = vol == MAX(vol, N) **绝对新高** ≠ vol >= MA * 1.5 **量能激增**
     - "价新高" = close == MAX(close, N) **绝对新高** ≠ close > MA * 1.05 **涨幅突出**
     - "趋势向上" = slope > threshold **斜率向上** ≠ close > MA **均线之上**
   - **修复方法**：要么改代码用严格的"字面对应"公式，要么改文档用准确的工程术语

3. **数据总线时钟命名残留自检**：
   - 列出所有时间相关变量名（`D` / `T` / `today` / `next_day` / `signal_date` 等）
   - 每个变量名必须**唯一锚定**一个具体日期语义
   - **禁用命名清单**（v1.3 → v1.7 演进教训）：
     - ❌ `D` / `T`（无时钟锚点）
     - ❌ `today` / `next_day`（无 regime 锚点）
     - ❌ `signal_date`（v1.3 残留，v1.7 应改为 `index_top_date`）
   - **统一命名规范**（v1.7.5）：
     - `index_top_date` = 大盘筑顶确立日
     - `topping_window` = 筑顶期窗口（index_top_date - 5 ~ -1）
     - `entry_date` = 个股验证入场日（index_top_date + 1）
     - `holding_window` = 持仓期（entry_date ~ entry_date + 20）
   - **DB schema 同步**：所有 `signal_date` / `expected_entry_date` 必须重命名为新规范

4. **模糊常量缺数学判别式自检**（每个"自然语言条件"）：
   - 列出所有"大多数" / "极少" / "偶尔" / "明显" / "强劲" / "温和" 等模糊词
   - 每个模糊词必须**强制量化**：要么改用数字（"≥ 60%"），要么改用绝对阈值（"≥ 3 天"）
   - **双条件兜底原则**（防边界绕过）：
     ```
     # ❌ 仅比率：5 窗口里 1 天高开但满足 60% 比率 → 边界绕过
     HIGH_OPEN_RATIO = 0.60
     
     # ✅ 双条件兜底：比率 + 绝对天数
     HIGH_OPEN_RATIO = 0.60          # 比率 ≥ 60%
     HIGH_OPEN_MIN_DAYS = 3          # 绝对天数 ≥ 3 天（防单日突发拉高）
     ```
   - **OOS 敏感性分析**：每个量化常量必须做 3 档对比（默认 ±20%），验证稳定性
