设计日内数据流时，**必须按"钟表时间"严格分段**，不能用"逻辑顺序"代替"实际时间"。每个 filter 调用前**必须验证"所需数据是否已生成"**。

**典型陷阱**（2026-06-24 PR-N Dual-Track 设计文档 v1.0 踩过）：

- 我把"09:30-09:32"写成单阶段流程：09:30:30 个股 filter（含"次日开盘 ≥ 7.8%"）→ 09:31:00 双轨分类器（含"5min ≥ 2%"判定）
- 09:31 距开盘才 1 分钟，5min bar **要到 09:35 才有完整数据**
- 30min 降级路径同样问题——30min bar 在 10:00 才完整，09:31 同样验证不了
- 用户原文："这不是策略分歧，是纯机械错误，必须改"+"实盘第一天就会因为拿不到数据而全部失败或报错"

**Why**：我设计时把时间当成了"逻辑步骤序号"（步骤 1、步骤 2、步骤 3...），而不是"真实挂钟时刻"。每个 filter 的可验证性必须**先**对"挂钟时间 + 数据可用性矩阵"做硬核对，再设计流程图。

**How to apply**：

1. **绘制数据可用性矩阵**（实施前必做）：
   - 5min bar：09:35 起每 5min 完整（09:35, 09:40, 09:45...）
   - 30min bar：10:00 起每 30min 完整（10:00, 10:30...）
   - 日线 OHLCV：D 日 15:00 后完整
   - 09:25 集合竞价：09:25 后可读
   - 指数日线：同个股日线

2. **设计流程时按"挂钟时刻"分段**：
   - 段一：09:25-09:30（盘前预热，只读昨日/集合竞价）
   - 段二：09:30:05（开盘后 5 秒，可读 open，不可读 5min）
   - 段三：09:35:05（**5min bar 首次完整**）
   - 段四：10:00:05（**30min bar 首次完整**）

3. **强制反退化测试**（已沉淀到 PR-N §6.1a）：
   - 在错误的时刻调用某个 filter → 必须抛 `DataNotReadyError`
   - CI grep 整个 codebase：`09:31` + `5min` 字面量**禁止同时出现**
   - 见 `tests/test_event_detection/test_data_availability_guard.py`

4. **下单时机判定**：任何"open + 头 N 分钟"双重验证的需求，**必须先 N 分钟 bar 完整后再下单**——哪怕逻辑上你"想要" 09:31 下单，实盘也得等到 09:35。
