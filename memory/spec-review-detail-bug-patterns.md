设计文档/技术规范在"整体逻辑严密"的状态下，仍会藏 **6 类细节 bug**——review 时必须用 6 类 checklist 排查，**不要被"宏观逻辑 OK"麻痹**。

**典型陷阱**（2026-06-24 PR-N Dual-Track 设计文档 v1.3 整体逻辑严密，但用户 anime 藏在细节里找出 6 处问题）：

| # | Bug 类型 | v1.3 案例 | 修复 |
|---|---------|---------|------|
| 1 | **流水线角色冲突** | 5min 同时承担"硬门槛（< 1% 剔除）"和"软门（失败透传）"两个角色 | 按数据可用性分层：硬底线 / 软区间 / 通过 三层 |
| 2 | **仓位/状态档位死代码** | `{0: 0.50, 1: 0.40, 2: 0.30}` 映射表里 0/3 在上层被截流，0.50 永不触达 | 删除不可触达的档位，保留 `{1: 0.50, 2: 0.40}` |
| 3 | **文字与代码逻辑不一致** | §4.3 Step 4 文字说"4 必须：MA5/连续跳空/30d 高/光头光脚"，但代码注释光头光脚在第 3 层 | 文字严格对齐代码注释 + 表格化定义 |
| 4 | **config.py 常量重复定义** | 同一行 `COND_BARE_CANDLE_SOFT = True` 出现 2 次（line 312/313） | 删一行，single source |
| 5 | **关键阈值常量缺失** | 30min 降级路径没有 anchor 阈值，运行时靠魔术数字 0.015 | 补 `ENTRY_30MIN_FALLBACK_THRESHOLD = 0.015` |
| 6 | **执行边界模糊** | PR-C 距 52w 高没说清楚属于 Global Pre-Filter 还是分类轴 | 显式声明"PR-C 是前置物理硬门槛，不进 10 个分类轴"，并画 Stage 1/Stage 2 分层图 |

**Why**：宏观逻辑（流程图 / 数据流 / 算法骨架）容易写对——因为有"框架性思维"驱动。**细节 bug 藏在局部**——一行常量、一个映射值、一段文字与代码的不一致——这些没有"框架性自检"覆盖，只能靠 6 类 checklist 强制排查。

**How to apply**（设计文档 review 必走 6 类 checklist）：

1. **流水线角色冲突自检**（每个数据/时间点一个角色）：
   - 列出每个数据源（如 5min bar）的**所有使用点**
   - 每个使用点必须**只能扮演一个角色**——硬门槛 / 软门 / 通过条件
   - 如果一个数据源被用于"硬剔除"和"软失败"两个目的 → **冲突**，必须拆为硬底线 / 软区间 / 通过 三层
   - 例：5min ≥ 2% 不可用时（09:31 数据未生成）→ 不能既"剔除"又"软失败透传"，应该"硬剔除（< 1%）+ 软失败透传（1-2%）+ pass（≥ 2%）"

2. **仓位/状态档位死代码自检**（每个映射表的每个 key）：
   - 对 `dict.get(key, default)` 形式，列出所有 `key` 的实际触发范围
   - 不在范围内的 `key` 对应的 value 是**死代码**
   - 例：仓位映射 `{0:0.50, 1:0.40, 2:0.30}` —— 但 failed_3==0 已 SKIP 进 TRACK_1，failed_3==3 已 SKIP（> 2 容忍），函数实际只接收 1/2 → `{0: 0.50, 2: 0.30}` 都是死代码

3. **文字与代码逻辑一致性自检**（每个文字描述 vs 对应代码注释）：
   - 文档里的"4 必须条件清单"必须**逐字**对应 `classify_track()` 代码里的硬门检查列表
   - 任一不一致 → 文档落后于代码，必须以代码为准改文档
   - 例：v1.3 §4.3 文字"4 必须：MA5/连续跳空/30d 高/光头光脚"——但代码里光头光脚在第 3 层（3 强化条件之一）→ 文档错

4. **config.py 常量去重自检**：
   - `grep -n "CONST_NAME =" config.py` 必须只返回 1 行
   - 多行 → 必有重复或层级错位，立即合并/删除
   - 实施期：CI 加 `assert len([line for line in open('config.py') if line.startswith('CONST_NAME =')]) == 1`

5. **降级路径 anchor 常量自检**（每个 fallback 路径）：
   - 任何 "X 不可用 → 降级到 Y" 的路径必须有 anchor 常量
   - 例：5min 不可用 → 30min 替代 → 必须有 `ENTRY_30MIN_FALLBACK_THRESHOLD = 0.015`，不能让运行时硬编码 `0.015`
   - 自检：`grep "0.0" *.py`（匹配魔术数字）→ 每条都对应一个常量定义才合规

6. **执行边界自检**（每个 filter / 分类器）：
   - 明示该 filter 属于"前置过滤（Global Pre-Filter）"还是"分类轴（Signal Classifier）"
   - 前置过滤：**决定是否值得入场**（仓位上限 / 物理硬门槛 / 风险排除）——不进分类轴
   - 分类轴：**决定信号强度**（Track 1 / Track 2 / SKIP）——决定仓位档位
   - 例：PR-C 距 52w 高 = 前置过滤（仓位上限）≠ 分类轴（信号强度）—— 混在一起会让仓位矩阵失去正交性

**类似陷阱**：
- API 文档：endpoint 列了 N 个参数，但实际 handler 只读 N-1 个 → 死参数
- 数据库 schema：列名重复定义 / 缺索引 / 缺外键约束
- 配置系统：默认值在多处被覆盖，到底哪个生效说不清
- 测试用例：断言写"≥ 55%"，但 baseline 是 54.9% → 断言几乎必通过，无意义
- CI/CD：脚本里写 `python3` 但 venv 用 `python` → 调用错路径
